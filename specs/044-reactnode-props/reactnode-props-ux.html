044 · ReactNode Props — UX Exploration

Exploring how users populate ReactNode props (like Badge's iconLeft and children) with HTML or other configured components. The key interaction: arm a component → click a prop drop target to assign it. This enables bottom-up composition with no nesting depth limit and no embedded component pickers.

2 — Drop Target States

The Component-mode drop target has three visual states: empty (default), receptive (a component is currently armed — teal glow, inviting click), and filled (a component has been assigned — shows mini thumbnail + name + clear button).

3 — Full Flow: Arm → Click-to-Set

The complete interaction for assigning an Icon component to Badge's iconLeft prop. The user works with two component rows — they don't need to leave the component list or use any picker.

1
Expand Badge — click Customize. See props. Switch iconLeft to Component mode. The drop target shows "Arm a component, then click here" (empty state).
2
Find Icon in the component list. Expand it, configure props: name="check", size="sm".
3
Arm the Icon — click the "Insert" button on the Icon row. It turns orange ("Inserting"). All visible ReactNode drop targets instantly glow teal (receptive state).
4
Click Badge's iconLeft drop target — the armed Icon is assigned. Drop target switches to filled state (mini thumbnail + "Icon" + ✕). Badge's ghost re-renders showing the composed result.
5
Done. Arm Badge itself and place it on the page — the agent receives: <Badge variant="primary" iconLeft={<Icon name="check" size="sm" />}>Active</Badge>

5 — Open Questions

Design decisions to resolve during implementation and testing.

Q1: Should the armed state clear after assigning to one slot?
Option A: Clear immediately — user arms Icon, clicks one drop target, done. Simple and predictable.
Option B: Stay armed — user can click multiple drop targets to assign the same Icon to several slots. Click the orange button again to disarm.
Option B is useful when you want the same Icon in both iconLeft and iconRight.
Q2: How does arming for "prop slot" vs "page insertion" work?
Today, arming a component targets the page (Insert/Replace). Now we're adding a second target: prop slots. Options:
Option A: Same "Insert" button serves both — armed component can be clicked on the page or on a prop drop target. Simpler, dual-purpose.
Option B: Separate buttons — "Insert" for page, "Use as Prop" for slots. More explicit, but clutters the row.
Option A feels cleaner: the user arms, then clicks wherever they want the component to go.
Q3: Do we need the mode tabs, or should Component mode be the default for ReactNode props?
Three tabs (Text / HTML / Component) may be overkill for simple props like children where users usually just want to type text. Options:
Option A: Start in Text mode, tabs available to switch. Familiar and non-disruptive.
Option B: Start in Component mode for non-children slot props (like iconLeft), Text for children.
Option C: No tabs — always show a combined drop target + text input. The text input doubles as simple text mode, and the drop target is always visible below it.
